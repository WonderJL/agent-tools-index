# Hook pillar

The hook pillar is the system's lifecycle automation: small, deterministic scripts that fire on a coding agent's editing events — before a write, after an edit, at the end of a turn. There are 4 authored hooks and no vendored ones. They are the safety rails and tidying chores that run automatically while an agent works, so a human doesn't have to remember them and an agent can't quietly skip them.

Each hook is a directory keyed by its id, holding a provider-neutral intent file (the authoritative declaration of its event, optional tool matcher, target script, and tier), a dispatcher script, and a short README. The intent is the canonical source; per-host adapters lower one intent into each agent's native config rather than re-authoring it per tool. Hooks are resolved and installed from this curated library through a dedicated CLI using a `hook@<id>` reference — mirroring `skill@<name>` and `subagent@<name>` — and are never hand-written into a host settings file or plugin, a boundary recorded in a decision record (ADR-0003 / ADR-0004). Two safety properties define the pillar. First, committed scripts stay dormant: each one no-ops on exit unless a per-machine opt-in file exists, so a hook that travels through git never runs on a teammate's clone until they explicitly enable it locally. Second, only one hook ever blocks. The categories are: secrets (blocking) — `block-secrets`, which refuses writes to dotenv and secret-bearing paths or content and is the *only* hook that exits non-zero; format/lint — `format-on-edit`; typecheck — `typecheck-on-stop`; and build check — `build-check-on-stop`. The stop-time checks report errors but never block the human.
